What is Coupling?

Coupling is the term used in computer science to indicate dependencies between software modules (i.e. classes). The higher the coupling is in a system, the less flexible the system would be. Consider the following example:

In figure A, class A depends on both class B and C. This suggests that if class C changes, then there has to be checks to see whether changes to both class A and class B are needed.

In figure B, class A depends on class B who depends on class C. This suggests that if class C changes, there has to only be one check to see whether class B needs to change - and, if it doesn't, a check to class A would not be necessary. Thus figure B has lower coupling.

What is Cohesion?

Cohesion refers to how related and focused the functionality of a software module is. A highly cohesive class contains methods that all pertains to providing the responsibilities of a class.

Consider a class that represents an animal in a software system to support the feeding of animals. The code below is an example of one implementation of the class with high cohesion:

// An animal in the aquarium.
class Animal {

// Construct an animal. We need to say what food the animal eats.
Animal(List<Food> acceptableFoodToEat) {...}

// What can this animal eat?
List<Food> getAcceptableFoodToEat() {...}

// When should the animal eat next?
Date nextTimeToFeed() {...}

// Remember what the animal last ate
void recordLastFeeding(List<FeedingRecord> foodEaten) {...}

On the other hand, consider the following low cohesion implementation:

// An animal in the aquarium.
class Animal {

// Construct an animal. We need to say what food the animal eats.
Animal(List<Food> acceptableFoodToEat) {...}

// What can this animal eat?
List<Food> getAcceptableFoodToEat() {...}

// When should the animal eat next?
Date nextTimeToFeed() {...}

// Remember what the animal last ate
void recordLastFeeding(List<FeedingRecord> foodEaten) {...}

// Print a photograph of a visitor at the aquarium
void printPhoto(Photograph photo) {...}

There is no reason why the Animal class would support the printPhoto method, so this implementation would have low cohesion.

Common Design Patterns

Singleton Pattern

The singleton design pattern is a design pattern that restricts the instantiation of a class to only one object. In this design pattern:

  • The constructor is kept private.
  • A public static operation getInstance() is defined.

An example of this would be the StopManager class from the term project. It is implemented as follows:

public class StopManager {
  private static StopManager instance;

  private StopManager() {
    this.stopMap = new HashMap<>();
    this.selectedStop = null;

  public static StopManager getInstance() {
    if (instance == null) {
      instance = new StopManager();
    }
    return instance;

Observer Pattern

The observer design pattern is a design pattern where the object, called the subject, maintains a list of its dependents, called observers, and notifies them automatically about any state changes.

Problems that the observer pattern solves:

  • A one-to-many dependancy between objects should be defined without making the objects tightly coupled.
  • When one object changes state, an open-ended number of dependent objects should be updated automatically.
  • One object should be able to update an open-ended number of dependent objects.
